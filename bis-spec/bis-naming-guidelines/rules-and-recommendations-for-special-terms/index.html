Rules and Recommendation for Special Terms

This sections contains rules related to special terms from the BIS upper ontology

🔗Model

Rule Model classes must always be suffixes with Model.

Example: FunctionalModel, PhysicalModel Exception: Model1d, Model2d, Model3d

Recommendation Model should be names based on what is being modeled or on what it contains what about perspective.

Example: BuildingModel or FloorsModel or Model2d or a combination such as Building2dFloorModel

We need more discussion on this waiting for imput from Shaun

🔗Element

Rule Abstract terms should only appear higher in the inheritance hierarchy than concrete terms.

Example: The term Element is used to suffix some abstract classes. Once Element is dropped from the name – because the class is concrete - it never comes back.

User-facing classes are normally concrete and we especially don’t want to add superfluous terms to those.

See: list of special terms

Recommendation User-facing classes should not be expressed in abstract terms

Example: Don’t use InformationElement when you can use Document

🔗Aspect

Recommendation Suffix with Aspect

🔗Type

Recommendation Use the term Type for real world types rather than Class or Kind

Example: FunctionalType rather than FunctionalClass or FunctionalKind

🔗Role

Recommendation

Don’t suffix with Role

Example: Resource, Asset rather than ResourceRole and AssetRole
