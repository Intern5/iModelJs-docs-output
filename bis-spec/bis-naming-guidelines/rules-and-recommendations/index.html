Rules and Recommendations

The following Rules and recommendations should be honored when devising BIS names[^1].

NOTE: The items below are not ordered by importance but rather around related concepts.

🔗General

BIS naming convention is deliberately more restrictive than EC naming: All BIS names are valid EC names but not all EC names are valid BIS names.

Rule Only use alphanumeric characters

Rule Do not use an underbars in names

Exception: Don't use underbars even if it aids readability: Use OrderLineItem not Order_LineItem

Rule Names may not start with a number

Example: 3dGeometry is not allowed

Rule Names may not differ only by case

Example: If there's already a name called Pricelist then you may not define a new one called PriceList

Rule Use PascalCase names

PascalCase means words are concatenated with the first letter of each word capitalized

Example: OrderLineItem

Rule Use United States Spelling

Example: Organization not Organisation

Recommendation Use the most widely accepted and descriptive name possible.

Consult the current thesaurus before creating new words.

Recommendation Use a single word that describes the entity

Example: Person rather than HumanBeing

Rule Don't add superfluous terms Don't use two nouns if one can equally express its meaning

Example: Vehicle rather than TransportationVehicle

For realtionships we often know the something of the target from the source: Use PhysicalElementIsOfType rather than PhysicalElementIsOfPhysicalType (because the element cannot be of any type other than Physical!)

Recommendation Do not add any unneccessary prefixes or suffixes to an entity

Especially terms like Object, Instance, Entity, and Property.

Example: Use Note not NoteEntity (for a class name) or NoteProperty (for a property name).

Recommendation Words should be less than 30 characters

Recommendation Use abbreviations and acronyms with care.

Use abbreviations and acronyms only where they are common and completely unambiguous. If the abbreviation/acronym is not well known, don't use it!

Try to limit the use to those that appear in the list of abbreviations and acronyms and don't uyse those listen in the "don't use section."

Two letter acronyms have both letters capitalized, provided that each letter represents a different word.

Example: UI ("User Interface") and IO ("Input/Output") because both words are capitalized, but Db because "Database" is one compound word.

Three or more letter acronyms have only the first letter in capital case.

Example: Html or Guid

Abbreviations of a single word have only the first letter in capital case.

Example: Id ("Identification")

Recommendation Use numbers with care

Avoid names such as One21 and Door2Door

A well-known exception is for terms 1d, 2d, and 3d (dimensions). These may be used to form new terms provided they don't break the rule stating that Names may not start with a number.

Example: Acceptable names are Spatial1d, Model2d

Recommendation Use special words with care. Specifically, make sure that the semantic meaning of the special term is well understood.

Example: List, Item, Set, etc... See list of special terms

If it isn't an Aspect, don't put Aspect in the name.

Rule Do not use New, Old, Tmp, or Temp

What seems "new" when you do it will become "old" when you make the next change.

Recommendation Avoid the use of 'Base' in class names

Example: use RasterModel instead of RasterBaseModel

Recommendation Avoid using the organization's name

If prefixes are needed, give preference to the domain rather than the organization.

Example: Use TransportElement rather than ExorElement

Recommendation Avoid using product names

Example: Avoid prefixes like eB, OpenPlant, STAAD, Dgn, ConnectedProject

Recommendation Avoid domain names in the term

Example: PlanningElement, ConceptualElement, PlantElement, PlantArea, and Area

Recommendation Avoid using version numbers or generation names

Example: V8iFile

Recommendation If you need to distinguish between an Entity Definition and an Entity Instance and there are no suitable words that can distinguish them, suffix the definition rather than the intance: user will work with instance more often than definitions

Example: Use AttributeDefinition for the definition and Attribute for the value.

Note: The ECProperty is an example where this recommendation is not being followed. There are no plans to change that. Sorry.

🔗ECSchema / BIS Domain Names

Rule All EcSchema/BIS Domain Names must be registered with BIS workgroup to avoid conflicts. This includes the aliases.

Example: BisCore (schema name) and bis See list of BIS Schemas

Rule BIM Schema aliases must be in lowercase

Rule BIM Schema aliases must be less than 7 characters long

🔗Class

Rule Use singular form

Example: File not Files

Rule When combining terms, arrange them in increasing order of significance

Example: CableCar, AnalogWaterMeter Exceptions: 2d and 3d due to syntax constraints See: names can't start with numerals

Recommendation Do not use prepositions such as Of, With, On, An, In, From, etc...

🔗Property (including “Navigation” Properties)

Rule Use the plural form of the name when the attribute is a collection, otherwise use the singular form.

Document.AddedBy (a document can be added by a single person) but can have many notes, so Document.Notes

Recommendation Try to create attributes with the same name as their underlying type

Document.Lock is of type Lock

Exception: When the entity has more than on attribute of that type, or when it aids in readability: Document.AddedBy instead of Document.Person

Recommendation Don't include the primitive datatype (int, string) with the name

Exception: Dates. Since Date is both a data type and a more meaningful noun, something like DateAdded would be acceptable but NoteBlob would not be.

Rule Don't include the attribute owner name in the attribute name

Example: Document.Id not Document.DocumentId and Person.Name not Person.PersonName

Rule Prefix Boolean names with Is, Has, Can, May

Example: IsUnderChange, HasOtherRevision, MayChange

Rule When the field represents a bit mask, use the plural form of the word

Example: StateHints, HasFlags

Rule For NavigationProperties, prefer to use the name of the related ECClass as the property name

If further qualification is needed, consider prefixing words from the role label.

🔗Relationship

Rule Use Source-Verb-Target

Example: ParentHasChild, OrganizationSellsProduct

In most cases, the "source/target" will simplybe the name of the object. In those cases, the addition of an adjective can be helpful: Child in ElementHasChildElement

Rule (Picking up on the previous rule) Don't use conjunctions. Do not use a singular noun or verb, even if it clearly defines a relationship. Always use noun-verb-noun

Example: Don't use Marriage or ManAndWoman; use PersonIsMarriedToPerson

Rule The relationship name should indicate the cardinality.

The source is always singular and the target indicates multiplicity.

Example: ElementOwnsChildElement (1:N), ElementHasLinks (N:N), PhysicalElementIsOfType (N:1)

Recommendation Use the full class names for source. For the target, use the 'role' within the relationship.

The target name may be abbreviated: especially when the source makes it obvious what the target name means.

Example: PhysicalElementIsOfType

Recommendation Use verbs that imply relationship "strength" with care

Especially with regard to EC where:

  • Owns or Contains implies embedding
  • Shares implies holding

See list of relationship strengths

[^1]: By name we include Domain, Schema, Namespace, Class, Properties etc.
