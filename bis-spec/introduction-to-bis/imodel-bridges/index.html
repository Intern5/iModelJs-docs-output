iModel Bridges

🔗Introduction

As of 2018, most data in iModels will have been generated in applications that do not use BIS for their native schemas. iModel Bridges exist for the purpose of transforming data from these other applications into iModels in the iModel Hub. Many Bridges can be associated with a single iModel and hence the resulting iModel becomes an aggregator of many sources of data.

Currently, iModel Bridges run within the context of ProjectWise. The source data for iModel Bridges usually resides within ProjectWise.

iModel Bridges need to carefully transform the source data to BIS-based data in the iModel, and hence each Bridge is written for a specific data source. Existing and planned bridges include:

  • Existing
    • AECOSim Building Designer
    • Microstation
    • RealDWG
    • Revit
  • Planned
    • ISM
    • OpenPlant
    • OpenRail Designer
    • OpenRoads Designer
    • PDMS
    • ProStructures
    • SmartPlant
    • Substation

Some key characteristics of iModel Bridges are:

  • Transformations of data are one-way from source to iModel.

  • Transformations may run regularly to keep the iModel up to date with the latest source data.

  • IDs and other data in the generated iModels are consistent through multiple transformations. This enables minimal and logical change sets within the iModel Hub.

  • Each iModel bridge "job" (combination of source data and bridge) generates data that is isolated from all other iModel bridge jobs' data. The resulting combined iModel is partitioned at the Subject level of the iModel; each Bridge job has its own Subject.

🔗Data Unification vs Data Faithful to Source Application

For each iModel bridge author, there will always be two conflicting goals:

  1. To transform the data in such a way that it appears logical and "correct" to the users of the authoring application (such as Revit).

  2. To transform the data in such a way that data from disparate authoring applications (such as Revit and ABD) appear consistent.

The appropriate balancing of these two conflicting goals is not an easy task. As the iModel ecosystem expands and users provide feedback, the optimal weighting of these objectives may change.

🔗Dynamic Schemas

Typically, BIS domain schemas are not adequate to capture all the data in the authoring application. To avoid losing data, iModel Bridges dynamically create application-specific schemas whose classes descend from the most appropriate BIS domain classes.

As iModel Bridges may (and usually do) run multiple times to keep an iModel up-to-date, the schemas created by previous executions limit the schemas that can be used by subsequent executions. To provide consistency and enable concise change sets, the Bridges add to the previously-defined schemas (creating new schema versions). This follows the general schema update strategy defined in Schema Versioning and Generations

Apply the 'DynamicSchema' custom attribute to the schema. This custom attribute can be found in the standard schema 'CoreCustomAttributes' and it enables us to programmatically detect dynamic schemas in the future.

🔗Display Labels

Wherever practical, the Elements generated from iModel Bridges should be identifiable through an optimal "Display Label".

As discussed in Element Fundamentals, the DgnDb stack generates the Display Label through the following logic:

  1. If the UserLabel property is set, it is taken as the Display Label.

  2. If the CodeValue is set (and the UserLabel is not set), the Code (fully scoped for uniqueness) is taken as the Display Label.

  3. If neither UserLabel nor CodeValue is set, then a default Display Label is generated from the following data:

    • Class Name
    • Associated Type's Name (if any)
    • 64-bit Element Id in standard user-facing format.

iModel Bridge data transformations should be written considering the DgnDb stack Display Label logic; UserLabel is the appropriate property for a Bridge to set to control the Display Label (CodeValue should never be set for anything other than coding purposes).

But what value should an iModel Bridge set UserLabel to? There are three goals to consider in the generation of UserLabels. Those goals, in priority order, are:

  1. Consistency with source application label usage.

  2. Consistency with BIS domain default labeling strategy.

  3. Uniqueness of UserLabels.

If the source application data has a property that conceptually matches the BIS UserLabel property, that value should always be transformed to UserLabel.

Each BIS domain author should provide a default Display Label strategy for each of its classes. This strategy should be be determined by the optimal user experience and typically will represent the consensus of Product Management for applications and services that are associated with the domain. Information that is typically considered in the domain's Display Label logic is:

  • Class Name

  • Associated Type instance

  • Key Property (e.g. pipe diameter) or Key Relationship (e.g. material name)

  • 64-bit Element ID in standard user-facing format
