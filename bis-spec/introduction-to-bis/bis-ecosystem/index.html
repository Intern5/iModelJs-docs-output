BIS Ecosystem

🔗Introduction

BIS is conceptual schema that by itself has little value. The value comes as applications and services that use the schema are built. As more applications and servces use the schema, the value increases more than proportionally. This chapter discusses some of the key technology in the BIS ecosystem.

🔗The DgnDb “Stack”

DgnDb is a cross-platform (Windows, Linux, iOS, Android) set of libraries for developing applications (desktop and mobile) and services. The DgnDb technology stack is built on SQLite and operates specialized SQLite database files that use the “DgnDb Format”. Details of the stack are defined elsewhere.

There is a largely 1:1 correspondence between BIS domain and DgnDb domain libraries; the DgnDb domain libraries provide special functionality to make working with the domain easier and less error-prone.

🔗The DgnDb Format

The DgnDb format is basically the SQLite format with a DgnDb-specific db schema that is designed to hold information that conforms to the BIS “conceptual schema.” The db schema is actually fixed, and all of the BIM information conforming to BIS is mapped into those somewhat generic tables.

Some of the ECProperties used in BIS contain BLOBs formatted using a “micro-format”, e.g. the Google Flatbuffer-based GeometryStream format holds geometric details, so that geometric concepts like arcs, bsplines, meshes, and toroids are defined by the GeometryStream micro-format, rather than as ECClasses. There are other micro-formats—some binary based, and some based on JSON encodings of information in a string ECProperty. The ECProperty’s “extendedType” attribute is used to denote the micro-format.

The fundamental concepts of DgnDb (e.g. “Model” and “Element”) are defined as ECClasses in BIS. The DgnDb stack will not work with non-BIS ECSchemas, and applications are not allowed to add their own SQLite tables… all information must be defined using ECClasses and persisted using the DgnDb/ECDb technology.

Files using the DgnDb format previously used a “.dgndb” file extension have recently started using a “.bim” file extension, and are called “BIM Files” (or occasionally just “BIMs”), though the file format is still officially “DgnDb Format”. [xxxxxxxxxxxx when is .iModel used? xxxxxxxxxxxxxxxxxxxx]

🔗The Nature of Information

This is an important aside that will be referenced in the definitions below, and must be understood in order to have a firm grasp of BIS and BIM.

What is “pure information”2 in the real world? We know it exists, but we can’t touch it.

For information to exist, it must be encoded into some physical media3 (e.g. a magnetic media). The encoding of the information into the media (e.g. the arrangement of the magnetic particles such that they carry the information) is known as a “carrier”4 of the information.

Let’s consider the example of a “poem”. It is “pure information”. There is only one instance of the poem “Jabberwocky” (by Lewis Caroll) in the universe. However, the existence of that information (the poem) depends on the physical world. That poem has to be “carried” by at least one instance of physical media, or else it will cease to exist. A human brain can be the “media” for a poem, with the arrangement of matter in the brain somehow “carrying” the poem. When I write down the poem, the arrangement of the ink on the paper somehow carries the poem. If there were only a blob of ink on the paper, it would not be carrying the poem.

Practically speaking, we tend to assign identifiers or names to the “pure information”, but (practically speaking) we mostly work with “carriers” of that information. We could separately identify individual “carriers” of information, e.g. the “merchant copy” of my credit card receipt information and the “customer copy”… and all of those carriers are encoded in some media.

As evidenced by the “merchant copy” example, the “carrier” is often colloquially referred to as a “copy” of the information, which is technically incorrect because the pure information is non-physical, and only one “instance” of it exists in the universe, so the “pure information” cannot truly be copied. When you “make a copy of” the poem “Jabborwocky” you don’t end up with two different poems named “Jabborwocky”, you end up with yet-another-carrier-of the one-and-only “Jabborwocky” poem. What is actually being copied is the encoding that “carries” the information in some media.

Generic Terms and Specific Implemenations

We define a few terms below (i.e. “BIM File”, “BIM Briefcase”, and “BIM Revision”) that are actually implementation-agnostic concepts, but because the DgnDb-based implementations were first, those unqualified terms are used to refer to the DgnDb-specific implementations of those concepts. As additional implementations appear, we give them more-qualified term, e.g. “BIM JSON File” and “BIM JSON Revision” or “BIM Cloud Storage”.

🔗BIM Repository

A BIM Repository is a named collection of "pure information" that conforms to BIS.

Each BIM Repository has a RepositoryId that is a 128-bit UUID.

Being “pure information”, there is no “implementation” of the BIM Repository concept. The implementation comes in when we start talking about “carriers” of the information of the BIM Repository… [xxxxx move or delete? Xxxx]

🔗BIM File (aka “.bim file”)

The generic term “BIM File” would refer to any formatted file that is a “carrier” of the information of a BIM Repository, but our default meaning (when you see the unqualified term) is to assume the is just a synonym for the DgnDb implementation, thus “BIM File” refers to a DgnDb-formatted SQLite file (typically with the .bim or .ibim file extension) that carries the information of a BIM Repository.

If I copy a .bim file, I’m copying a carrier of a BIM Repository (I’m not copying a BIM Repository.).

🔗BIM Briefcase

A BIM Briefcase is almost a synonym for “BIM File”, except that it connotes that the file might only carry a subset of the information of the BIM repository, e.g. it might be “filtered copy” of the “Master” BIM File (that carries the complete and most-reliable information of the BIM Repository.)

Today, we don’t actually support “filtered copies” of BIM Files, so “BIM File” and “BIM Briefcase” really are synonyms. We do intend to support filtering, someday, e.g. to be able to give a sub-contractor only a subset of the information of a BIM Repository.

Again, there may eventually be non-DgnDb implementations of the “BIM Briefcase” concept, but the unqualified “BIM Briefcase” term refers to the DgnDb-specific implementation.

All BIM Briefcases will contain the RepositoryId of the BIM Repository whose information they are carrying, as well as a 24 bit integer BriefcaseId that is assigned by the BIM Server (see below) and is unique only within the BIM Repository.

🔗BIM Collaboration Services (aka iModelHub)

BIM Collaboration Services (BIM CS) is a set of cloud (and/or on-premise) services with http-based REST APIs. As of this writing, the implementations of several of the services are actually lumped together into a “BIM Server” service, but eventually they will be broken out into services such as:

  • BIM Repository Management Service (BRMS): Upload/download of BIM Files, push/pull of BIM Revisions, management of BIM Briefcases (and their ids), and (in the future) branching of BIM Repositories

  • BIM Lock Management Service (BLMS): Locking bis:Models and bis:Elements.

  • BIM Code Management Service (BCMS): Generating, validating, and registering Codes

  • BIM Change Event Service (BCES): Notifies subscribers of any change events in BIM CS.

  • And more in the future… BIM Live Web Access Service (BLWAS), BIM Published Web Access Service (BPWAS), etc.

Collectively, the services can be thought of as “Mercurial for BIM Repositories”.

Clients of BRMS can download a BIM Briefcase (BIM File) of any BIM Repository that the server is managing, and then pull and push incremental changes to the BIM Repository in the form of BIM Revisions (see below).

The BCES allows clients to subscribe to events like “A new BIM Revision has been pushed” and then pull the BIM Revision into their own BIM Briefcase. The clients may be desktop or mobile apps or other cloud services. For example, a mobile app like Navigator Mobile can keep large BIM Briefcases up-to-date without requiring repeated large file downloads.

The BLMS manages pessimistic locking (which is optional). In the future, BIM Clients will also support optimistic concurrency with conflict-resolution.

In the future, the BRMS will support allowing “filtered” BIM Briefcases.

🔗BIM Revision

A BIM Revision (in general) is any formatted xxxxxxxxxxxxxxxxxxx carrier of information xxxxxxxxxxxxxxxxxxxxx about the changes required to “revise” a BIM Repository from a pre-revision state to a post-revision state. As usual, we reserve the unqualified term to refer to the DgnDb-based implementation, though we have already implemented a JSON-baesed “BIM JSON Revision” as part of some synchronization infrastructure.

In DgnDb BIM Revisions are binary blobs used as transportable packages of changes made to a BIM File. The BIM Revision is a thin wrapper around a SQLite ChangeSet, which records changes to a BIM File in a compact format.

When you “pull from” and “push to” the BIM Repository Management Service, you are pushing and pulling BIM Revisions. The “pull before you push” rule will be enforced.

🔗BIS Generations

A BIS Generation is a series of BIS releases in which only "minor changes" to schemas are allowed, where “minor change” is defined as any change that does not break compatibility with software written against any version of BIS within that Generation. Minor changes can include very significant additions (but no deletions) – including adding new domains - so a Generation is not as constricting as might initially be assumed.

For the DgnDb technology stack, the BIS Generation approach gives one huge benefit:

Software written for any version of BISwithin a Generation can work with .bim files written by software written for any version of BIS within the same Generation.

This flexibility largely removes the need for synchronized product releases and has the potential to vastly simplify the user’s product upgrade complexities. If the user stays within a BIS Generation, all the products and services should work together.

See xxxxxxxx for a detailed description of BIS Generations and the motivation for them.

🔗BIS Outside of DgnDb

BIS is not only intended for the DgnDb stack; BIS is intended to be the conceptual schema for all Bentley’s applications and services. The use of the same conceptual schema will allow all the facets of the BIS (and Bentley) ecosystem to interoperate. The expectation is that all new REST APIs exposing user-domain concepts like “user”, “asset”, “project”, “enterprise”, “major infrastructure”, “document”, “transmittal”, “issue”, “deliverable”, “rendition”, “work package” etc. will need to conform to BIS (meaning that there will need to be BIS ECClasses describing the resources exposed by those services and the JSON should use the semantics defined by BIS and a JSON format that is compatible with the overall BIM ecosystem.
