Element Fundamentals

🔗Introduction

A bis:Element is a record in a BIM Repository that represents some entity in the real world, e.g. pumps, beams, contracts companies, processes, requirements, documents, persons, etc. In constrast to DGN V8 technology, Elements are not inherently graphical, and very many will contain no graphical information.

🔗Identifiers in DgnDb

TODO: Need to move this to a different section as it applies to all “entities” (elements, aspects, models)

Every kind of record in a BIM Repository (e.g. ECSchemas, ECClasses, Models, Elements etc.) has an Int64 repository-specific identifier that is unique only within the BIM Repository (including across all “briefcases” of the BIM Repository.)

In the case of a DgnDb-implemented BIM Repository, this ID is generated by concatenating the BriefcaseId of the BIM Briefcase in which the record is “born” with a sequence that is specific to that BIM Briefcase. Note that though this ID essentially encodes information about the BIM Briefcase in which the record was first created, its ID does not change when it is copied into another BIM Briefcase of the same BIM Repository—it is not a “Briefcase-specific-Id” it is a “Repository-specific-id.

🔗Element Identifiers

Elements may have multiple identifiers, which are for identity in different scenarios:

  • Every Element has a CodeScope, CodeSpec, CodeValue triplet which define a human-readable unique identifier satisfying a coding (or “tagging”) system. These coding properties may not be defined for some Elements.

  • The ElementExternalKey aspect can be used to define a secondary coding value. An Element may have multiple ElementExternalKeys.

  • Every Element has a FederationGuid which can be used to identify corresponding Elements across technology stacks. This GUID value is typically not assigned until required.

  • In DgnDb, every Element has a 64-bit identifier which uniquely identifies that Element across all coordinated iModels.

🔗Code

Each is:Element in a BIM Repository has a “Code” that represents an identifier in the real world. Examples of identifiers that codes can represent include an Asset Tracking Number, Tag Number, Serial Number, VIN Number, Social Security Number, Document Tracking Number, Contract Number, Project Number, RFID, etc. When the identified entities are physical in nature, they will often have their “code” physically affixed to them (as in the VIN Number on a car or the Manufacturer’s Serial Number on a guitar).

Each Code consists of a CodeScope and a Value and is governed by a CodeSpecification (described below).

Though all bis:Elements have a CodeValue ECProperty, BIS allows each specialization of bis:Element to override the DisplayLabel of the CodeValue ECProperty, e.g. to label it (in the GUI) as “Name” or “VIN” or “SSN” or “Tag” etc. as appropriate.

Often the value of the CodeValue property will be used as a display label for the record to which it belongs. Every ECClass has a “display label rule” to indicate how display labels for their ECInstances should be generated, and (when a record has a non-null code) the rule is generally to display either the CodeValue or the CodeNamespace concatenated with the CodeValue.

🔗CodeAuthority

TODO: terminology change to Code Specification and Code Service

A “code authority” is a system in the real world (implemented behind an API) that is capable of generating new Codes within a given scope and using a given coding scheme. In the case of DgnDb-based BIM, code authorities are implemented as a C++ class (a subclass of DgnAuthority) but some of those DgnAuthority implementations can call the HTTP-based REST API of a CodeAuthorityService that provides the code-generation logic.

We mentioned previously that every Code is governed by a CodeAuthority. This is modeled using a relationship to a CodeAuthorityRecord.

Elements representing real-world entities that don’t have a meaningful real-world identifier, e.g. a piece of baseboard, a pile of dirt, an average bolt, will be assigned the “NullCode” authority and will have null for its CodeValue.

There is also a facility for generating “temporary Codes” when disconnected from CodeAuthorityServices. The temporary Codes get replaced by real ones when a connection is re-established.

Generally speaking, a CodeValue should not be a GUID, which is typically not a real-world identifier, but a repository-specific-id (even if it is used by multiple repositories.)

🔗AlternateCode Aspect

A bis:AlternateCode Aspect also represents an identifier in the “real world”, but handles a case where there is more than one for a given element, e.g. my laptop has both a Manufacturer’s Serial Number (the Code) and a Bentley Asset Tracking Number (the AlternateCode).

🔗UserLabel

Every bis:Element has a UserLabel ECProperty, which can be null, and is not really a formal identifier. Note that the display label used in the GUI may or may not have the same as the value of the UserLabel ECProperty, depending on the display label rule associated with the bis:Element’s ECClass.

In general the UserLabel should left for the user to enter and not be programmatically generated. In cases where it is appropriate to programmatically generate a UserLabel expect that a user may modify it in the future.

For Elements with a non-null CodeValue, the UserLabel may just serve an alternate “friendlier” name in a GUI, e.g. the Room with CodeValue=R-134 might have UserLabel=”The Big Kitchen” or UserLabel=”Bhupinder’s Office”

For Elements with a null CodeValue, the UserLabel can provide some meaningful display label, even if it is not strictly controlled by some CodeAuthority.

For Elements with null CodeValue and null UserLabel, we are forced to generate some other display label, e.g. the display label of the bis:Element’s ECClass concatenated with a lexical representation of the bis:Element’s ID

🔗Elements and Models

Each Element “lives in” a single Model. The Model “owns” the Element and the concept of cascading deletes applies. The Model provides the context for the Element and without that context the Element loses its meaning.

Every Model models some Element (that is not contained in that Model). This is a key principle of BIS that enables it to fulfill its mission of supporting modeling of reality from multiple perspectives (including multiple granularities) in a single BIM Repository. It also results in a coherent and predictable structure in every BIM Repository, where all information is clearly traced back to a single root “Subject” for the entire BIM Repository. For more detail, see section….XXXXXXXXXXXXX.

There is one exception to the “Every Model models an Element” rule: There is exactly one RepositoryModel in every BimRepository which does not Model another Element (at least not another Element in the same BIM Repository). This RepositoryModel is at the top of the Model/Element hierarchy. We will see later how it may model an Element in some other BIM Repository.

🔗Elements and Child Elements

An Element may have child Elements. Both the parent and the children must live in the same Model. The parent-child relationship implies ownership and cascading deletes. This is discussed more in the Section Assemblies and Groups.

🔗Elements and Locking

Element provide the finest grained locking that is available in BIS. Locking is not provided for individual properties or aspects.

🔗GeometricElement

GeometricElement is used to represent any real world entity that we chose to model (in our BIM Repository virtual world) as inherently being geometric—such that BIM software will be called upon to display it graphically. This includes physical entities (such as electrical panels, beams and pumps) as well as drawing graphics (lines, arcs, circles, and annotations like text, plots, etc) in geometric context). Merely having a geometric property is not necessarily enough to merit an entity being represented by a GeometricElement; for example, a steel section shape is an InformationElement, not a GeometricElement.

The GeometricElement hierarchy defined in the BIS Core is broken down as shown in the figure above.

Brief descriptions for the classes in the 2D branch are:

  • GeometricElement2D – An Element that is inherently 2D in nature.

  • GraphicalElement2D – xxxxxxxxxx.

  • DrawingGraphic – A GraphicalElement2D that is intended to be placed on a Drawing.

Brief descriptions for the classes in the 3D branch are:

  • GeometricElement3D – An Element whose representation in BIM is inherently 3D in nature.

  • GraphicalElement3D – A 3D Element that is not relevant to real-world 3D space, such as a 3D bar chart. These Elements could be positioned in real-world 3D space (in a heads-up display, for example).

  • SpatialElement – An Element representing an entity that exists in and is relevant to real-world 3D space.

  • PhysicalElement – A SpatialElement representing a real physical entity, i.e. one that has mass.

  • SpatialLocationElement – A SpatialElement representing some defined point, curve, surface, or volume in the real world. Spatial locations are not physical, and have no mass. Examples include a property line, a zoning envelope, an alignment, or a gridline.

🔗Mixins

Mixins play a key role in supporting cross-discipline coordination in BIS.

In the future, EC will be enhanced to have a formal interface feature, and BIS will take advantage of that capability. For the short term (BIM02) EC Mixin classes, custom attributes and schema validators will provide similar functionality.

A Mixin is an abstract EC class that follows a strict set of requirements that are intended to provide the desired functionality while minimizing confusion and implementation costs. The requirements may be relaxed in the future if use cases supporting relaxation are found. The requirements for a Mixin class are as follows:

  1. Mixins are abstract EC Entity classes.

  2. Mixins may have 0 or 1 base classes. If there is a base class, it must be another Mixin.

  3. Mixins must have the IsMixin custom attribute defined.

  4. Mixins may not override an inherited property.

The Mixin custom attribute is defined as follows:

<ECCustomAttributeClass typeName="IsMixin" description="Applied to abstract ECEntityClasses which serve as secondary base classes for normal ECEntityClasses." displayLabel="Is Mixin" appliesTo="EntityClass" modifier="Sealed" >
    <ECProperty propertyName="AppliesToEntityClass" typeName="string" description="This mixin may only be applied to entity classes which derive from is class.  Class Name should be fully specified as 'alias:ClassName'" />
</ECCustomAttributeClass>

A class incorporates a Mixin by declaring the Mixin class as a base class. An Mixin is never a the first base class listed (that position is reserved for the “real” base class). A class may implement multiple Mixins, but may only have one “real” base class. Classes that incorporate Mixins must also follow strict rules:

  1. The incorporating class must descend from the AppliesToEntityClass defined in the IsMixin custom attribute.

  2. The implementing class must not have an EC property (including inherited properties) that has the same name as a Mixin property.

Mixins can be used as relationship endpoints.

The rules for Mixins and the classes that incorporate Mixins will be enforced by two mechanisms:

  1. The EC Schema Editor will validate BIS schemas as they are edited.

  2. An automated (likely Firebug) build will continually check all BIS schemas for conformance.

By convention, interface-like Mixin classes have class names with an “I” prefix. Interface-like Mixins satisfy one or more of the following criteria:

  1. The Mixin is expected to be used as a relationship endpoint.

  2. The Mixin is expected to be used by code or logic that will not know which class has incorporated the Mixin.

  3. The Mixin has no properties and is used as a “marker” class.

🔗Assemblies

An Element can optionally have a (single) “parent” Element. The parent Element must be in the same Model as the “child” Element. A hierarchy of Elements can be created through these parent-child relationships; all the Elements in a hierarchy will reside in the same Model. An Element that is a parent, can be considered an “assembly” of all of its children. As there can be multiple levels in the hierarchy, there can be assemblies of assemblies. Circular networks of parent-child relationship are not allowed.

Assemblies imply ownership and cascading deletes – when the parent Element is deleted, all of its children are deleted as well.

Assemblies tend to fall into 4 behavior patterns:

  • The assembly is purely an aggregation; the primary role of the parent is to group the children together (example: a rebar cage collects various reinforcing bars together)

  • The assembly is an elaboration; properties (or other information) in the parent allow the children to be generated (example: a pipe support generates a pipe clamp, a beam clamp and a threaded rod)

  • The assembly is an item with modifications; the parent defines a base that the children modify (example: a BeamCope modifies a Beam).

As an Element can only have a single parent, it is important that our domains and applications coordinate to ensure there is no conflict over which parent an Element should have. The best general rule to coordinate parenthood is that the domain author determines which Elements (if any) can be parents of it’s Elements. Elements will generally fall into 4 categories:

🔗IParentElement and ISubModeledElement

This section tries not to repeat the material in Chapter 7. Model Fundamentals and Chapter 10. Model Hierarchy. If you have questions about how Elements breakdown into Models, you may want to skim those chapters.

There are two mixins that declare and define key behaviors of an Element:

  • IParentElement – this Element can be a parent Element

  • ISubModeledElement – this Element can have an associated breakdown Model

The Element class does not include either of these interfaces, so by default no Element can have children or can be broken down into a Model. Many Elements have no need for child Elements or breakdown Models and will therefore use neither of these interfaces; a Bolt class is a example of a class that requires neither.

These mixins are expected to be mutually-exlusive. No class that implements IParentElement will also implement ISubModeledElement (if potential exceptions arise, they will be considered). The reason these two interfaces are considered mutually-exclusive is that the two concepts (having children and having a model) are both means of breaking down something into more detail; using two different breakdown methods simultaneously could result in confusion and double-counting.

It should be noted that these mixins can be added to a class in a minor schema update that will not break generation compatibility, so there is no need to implement them until they are needed.

🔗Parent-Child Relationships

All Element parent-child relationships descend from ElementOwnsChildElements, which is abstract and cannot be used directly. There are four abstract specializations of this relationship that are frequently (but not always) used to clarify the relationship of the parent and child:

  • PhysicalElementEAssemblesElements – used to indicate that the child Elements are aggregated into the parent Element, whose geometry is entirely an aggregation.

  • PhysicalElementModifiedByElements – used to indicate that the child Elements modify (physically) the parent Element. The child Elements may be additive or substractive. Holes are a example.

  • ElementEncapsulatesChildElements – used when the child Elements represent internal data that is not typically exposed to user or useful outside of the parent Element’s context

🔗Groups

There is no formal “group” concept in BIS. Groups are discussed here to illustrate how they are different from assemblies. While assemblies are comprised of a strict parent – child hierarchy of Elements, other Element-to-Element relationships may defined groups of Elements, which may or may not be in the same model. The term “group” would generally not be used for an assembly.
