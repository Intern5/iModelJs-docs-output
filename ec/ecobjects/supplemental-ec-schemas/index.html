Supplemental ECSchemas

Supplemental schemas allow ECCustomAttribute instances to be moved outside of the primary schema. The primary schema is combined with the supplemental schema at load time so all custom attributes are available at runtime. A supplemental schema can only add or override custom attributes coming from the primary schema, they cannot add additional classes or properties or in any other way change the schema.

This feature is helpful for:

  • Reducing clutter in the primary schema

  • Grouping Custom Attributes by type

  • Grouping Custom Attributes that vary together

  • Applying different sets of custom attributes to the same primary schema depending on the situation.

Supplemental schemas use the same format as a normal schema but have several additional pieces of metadata and additional restrictions.

🔗Additional Supplemental Schema Metadata

Supplemental schemas must apply the 'SupplementalSchemaMetaData' custom attribute at the schema level. This custom attribute has two required pieces of information:

  • Back reference to the primary schema. Name and version must exactly match the primary schema

  • Precedence which determines what schema wins when there is a custom attribute conflict

🔗Additional Restrictions for Supplemental Schemas

  • The supplemental custom attributes must be applied to 'dummy' classes and properties that match the classes and properties of the primary schema. NOTE: A supplemental schema may override a base class's property so that it may apply a custom attribute to the derived classes property.

  • A supplemental schema should only create 'dummy' classes and properties for the classes and properties that it will apply custom attributes to.

  • A supplemental schema should not give 'dummy' classes base classes. Class hierarchy serves no purpose in a supplemental schema. Adding base classes can result in reduced performance and can cause issues not possible without them.

  • All properties should be of type 'string'. Types serve no purpose in a supplemental schema. Specifying a struct type in a supplemental schema can result in reduced performance and can cause issues not possible if built in primitives are used.

  • A supplemental schema may not define ANY classes not in the primary schema. This includes custom attribute classes.

  • A supplemental schema must conform to the following naming convention: _Supplemental_[_]. For example a supplemental schema with the Purpose 'StandardClassIdentification' for the primary schema ‘BentleyECM’ would be called 'BentleyECM_Supplemental_StandardClassIdentification'.

    • The [_] component is not required, it is allowed because there are situations where there may be multiple supplemental schemas for one purpose.

    • Because the Supplemental schema name includes the Purpose, Purposes may not contain any characters that can not be included in a schema name. Also, for parsing reasons a Purpose may not contain underscores (_).

    • NOTE: This requirement allows Supplemented schemas to be loaded automatically and supplementation must be done manually.

🔗Supplemental Schema Precedence

Supplemental schema precedence allows supplemental schemas to be split into multiple levels, reducing the likelihood of a conflict and providing information about the custom attributes in the schema. The Precedence is an integer value ranging from 0 to 699, higher values denote higher precedence. Custom attributes with higher precedence override ones with a lower precedence. The Primary schema has a precedence between 199 and 200 so schemas with a precedence from 0-199 can have their custom attributes overridden by the primary schema and schemas with a precedence from 200-699 can override the custom attributes on the primary schema. These two broad ranges of precedence are split into 7 levels to guide the user's choice of precedence values. The following list shows the precedence levels and their ranges.

  • Global (0-99)
  • Application (100-199)
  • Primary Schema
  • Customer (200-299)
  • Site (300-399)
  • Project (400-499)
  • Discipline (500-599)
  • User (600-699)

Remember that under most circumstances supplemental schemas may not have the same precedence if they apply the same custom attribute class to the same IECCustomAttributeContainer (IECSchema, IECClass, IECProperty). In some cases custom attributes can be merged so it would not necessarily create a conflict. An example of this is the custom attributes for defining Unit information in the Units Framework. These custom attributes are merged using a CustomAttributeMergingDelegate (see ‘Gaining more control of the SupplementedSchemaBuilder' for more information).

During supplementation only custom attributes defined directly on a container are taken into account. Custom attributes defined on a base class are not checked during the supplementation of the derived class. This means that a custom attribute from a schema of lower precedence than the primary will be applied even if the primary class's base class defines the same custom attribute. The only time that the supplemental custom attribute will not be applied is if the custom attribute is assigned directly to the container or applied to the container from a higher precedence schema.
