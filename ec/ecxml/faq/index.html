FAQ

This section elaborates on some of the design decisions behind ECXML. One critical design goal underlies most of the design decisions: that ECXML is an implementation of an intermediate "information model" designed to facilitate integration between JSpace, ODS, USpace, XML w/ XML Schemas, and the .NET type system.

🔗Why not just use vanilla XML with W3C XML Schemas?

The reason for there being an ECXML, rather than relying on many flavors of domain-specific XML, is that ECXML defines a common "information model" into which to transform the data, for example, a model that is more limited than vanilla XML, or a model in which data from many different application domains can be integrated and managed.

Normally, when converting an application's data to XML, minimal conceptual transformation is done, e.g. TriForma "Parts and Families" probably translates to something like TFPart and TFFamily, with TFParts nested in TFFamilies. One could translate JSpace, ODS, USpace, .NET Classes, etc. to/from XML, but when writing software to integrate all of these, one is still left with an N*N problem of transforming between the N different XML formats—you have only delayed the resolution of conceptual mismatch among systems.

Translating to ECXML requires a conceptual translation into a fairly rigid "information model" of classes, structs, properties, custom attributes, etc. When you’re done, however, your users can leverage information management applications written around that "information model".

The Publishing use case is perhaps most compelling. When an "intelligent" plot or published image of a TriForma drawing is created, how will the non-graphic data and metadata accompanying the graphics be stored? One solution is for every application to have their own format. Even if they are all XML formats, each of the browsing clients for the published drawings would need to understand each of the formats. It is much simpler to define one format, which we believe can reasonably capture the relevant data and metadata, and give the applications control over how the conceptual mapping into this format is done.

🔗Why have separate ECProperty, ECStructProperty, and ECArrayProperty

There are structural differences in each of these flavors of properties, and having them as separate elements will make handling of them more convenient. In a C# implementation, they would map to different classes. This improves our ability to validate ECSchemaXML and simplifies logic for handling arrays and structs with XSLT or program logic, since these structural variations are distinguished by their XML Element Tags.

🔗Rather than a special ECSchemaXML format, why not use W3C XML Schemas with xsd:app-info elements for additional metadata?

W3C XML Schema xsd:app-info elements provide M2 level metadata and above for XML Schemas—they play the exact same role as custom attributes in ECXML or .NET. We considered not having a special ECSchemaXML format and instead just using raw XML Schemas "decorated" with app-info to express the additional metadata we need. We decided against it because:

  • The XML Schema "information model" is not restrictive enough... not as restrictive as ECSchemaXML's

  • It is easier to keep the "information model" of ECXML/ECObjects clearly in mind if the semantics of the schema document are those of ECXML, not those of XML Schema, which has a much looser "information model."

  • We proved that we can generate XSDs from our ECSchemaXML documents, giving us the relative semantic simplicity of ECXML while still being able to use domain-specific XSDs for validation.

  • We could enhance our XSD generation to take the ECSchemaXML that generated the XSD elements and apply it to them as app-info...

🔗How does ECXML relate to ECObjects?

ECXML is a persistent format for ECObjects that is shared between both the C# and C++ implementations.

ECXML does not require the use of either the .NET or C++ implementations of ECObjects.
