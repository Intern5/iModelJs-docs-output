ECInstanceXML

We use an encoding that is largely compliant with the recommended SOAP Encoding for objects.

🔗ECInstance

🔗Element Name

The name of the ECClass of the ECInstance.

🔗Attributes

instanceID

An InstanceId is a unique identifier for an instance of a specific class in a repository. InstanceIds must be unique per-class-per-repository, meaning that in a repository, there may be no duplicate ids among any instances of a given class. InstanceIds may take many forms and may be changed by a repository when the instance is committed. InstanceIds are required for instances in a repository or when added to a relationship. An IECKey encapsulates all identifying information for an instance given a repository. It can be used to represent an IECInstance and contains schema name, class name and the instances id.

🔗Example

<Employee instanceID="434">
  <EmployeeNo>97</EmployeeNo>
  ...
</Employee>

🔗ECValue

🔗Element Name

The name of the ECProperty whose value this represents.

🔗Attributes

None, though incoming SOAP might have an attribute like xsi:type=”xsd:int”. We would ignore these and would not generate them ourselves. We determine the type of the value by looking up the property name in the relevant ECSchemaXML.

🔗Example

<EmployeeName>John Doe</EmployeeName>

🔗ECStructValue

🔗Element Name

The name of the ECStructProperty whose value this represents.

🔗Attributes

None, though incoming SOAP might have an attribute like xsi:type="MyStruct". We would ignore these and would not generate them ourselves. We determine the type of the struct value by looking up the property name in the relevant ECSchemaXML.

🔗Example

<Address>
  <Street>101 Main St.</Street>
  <City>Smithville</City>
  <State></State>
  <Zip>43343</Zip>
</Address>

🔗ECArrayValue

🔗Element Name

The name of the ECArrayProperty whose value this represents

🔗Attributes

The soap encoding indicates that an attribute SOAP-ENC:arrayType="{typeName}[{size}]", indicating the type and size of the array, be included on the array element. This is seemingly inconsistent with the specification that the xsi:type attributes are optional everywhere else, if the type info can be found in the XML Schema. The SOAP generated for Microsoft Web Services also omits this extra type info.

Furthermore, in SOAP 1.2, the recommended array encoding allows specification of the array item type and array size to be optional.

Therefore, we have also decided to omit use of a SOAP-ENC:arrayType attribute.

🔗Sub-elements

For primitive arrays, use Microsoft Web Services flavored SOAP, which uses an element name the same as the typename in the xsd namespace without actually referring to the namespace. For example, rather than subelements like xsd:string, only is used, however, the XML Schema declares that the value in is of type xsd:string.

For arrays of structs, subelements will be named according to typeName of the struct.

🔗Example

<Hobbies>
  <string>reading</string>
  <string>fishing</string>
  <string>tennis</string>
</Hobbies>

<PriorAddresses>
  <Address>
    <Street>123 Main St.</Street>
    <City>Smithville</City>
    <State></State>
    <Zip>43343</Zip>
  </Address>
  <Address>
    <Street>987 High St.</Street>
    <City>Smithville</City>
    <State></State>
    <Zip>43343</Zip>
  </Address>
</PriorAddresses>
